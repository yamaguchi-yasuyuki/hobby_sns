# システム要件定義書

## システム概要

**MITORI（読むだけSNS）** は、40代男性のSNS未使用者を主なターゲットとする、完全受信型のライフスタイルビジュアルフィードアプリである。「発信しなくていい」をコアコンセプトとし、投稿・フォロー・いいねといったSNS的な義務を一切排除する。ユーザーはキャンプ・温泉・サウナ・バイク・釣り・古着・ウイスキー・国内旅行などのジャンルを選択するだけで、自分のためのビジュアルフィードが生成され、気になったコンテンツをコレクションに貯めていける。X・note・YouTubeなどの外部コンテンツへの導線を持ちながらも、アプリ内では静かで上質な閲覧体験を提供する。スマートフォン・タブレット・PCのすべてに対応したレスポンシブWebアプリとして実装し、MVPでは手動キュレーションによるコンテンツ品質の担保を優先する。

1. **システムの目的**
   - 解決する課題：40代男性SNS未使用者が「発信しなければならない」という参入障壁なしに、趣味の良質なビジュアル情報を受信できる場を提供する
   - 目標とする成果：週1回以上の継続的な利用と、コレクション機能による習慣化
   - 期待されるビジネスインパクト：趣味関連スポット・商品への送客による提携収益の創出

2. **主な機能**
   - コア機能：ビジュアルフィード表示、ジャンルフィルター、「気になる」保存、コレクション管理
   - 差別化機能：投稿機能の完全排除、「〇人が気になっています」の人数表示のみの社会性、外部コンテンツへのシームレスな導線
   - 付加価値機能：フィードのパーソナライズ（Phase 2）、コレクションカテゴリ分け

3. **提供する価値**
   - ユーザーへの価値：発信プレッシャーゼロで趣味の情報収集ができる、自分だけのコレクションが育つ充実感
   - ビジネスへの価値：趣味関連企業との送客提携、ニッチで購買力のある40代男性層へのリーチ
   - 社会的価値：SNS疲れ・デジタルウェルビーイングへの貢献

4. **実現方法の概要**
   - 主要テクノロジー：Next.js (App Router) + TypeScript + Supabase + Clerk + Vercel
   - 重要なインテグレーション：外部コンテンツ（X・note・YouTube）へのリンク管理、アプリ内ブラウザ的なUI
   - スケーラビリティ：Vercel + Supabaseによるサーバーレス構成でスケールアウトに対応

## 機能要件

### 1. コンテンツフィード
| 機能 | 詳細 |
|------|------|
| ホームフィード表示 | カード型グリッドレイアウト。画像・タイトル・ジャンルタグを表示。無限スクロール対応 |
| ジャンルフィルター | キャンプ・温泉・サウナ・バイク・釣り・古着・ウイスキー・国内旅行の8ジャンル以上。ホームフィード上部のタブで切り替え |
| カード詳細画面 | 大画像・説明文・出典情報・「〇人が気になっています」・「気になる」ボタン・外部リンクボタンを表示 |
| 外部リンク遷移 | アプリ内ブラウザ（WebView相当）でX・note・YouTube等に遷移。ブラウザを閉じると元の画面に戻る |
| 「〇人が気になっています」 | コンテンツごとの保存人数の集計値を表示。個人特定不可 |

### 2. コレクション管理
| 機能 | 詳細 |
|------|------|
| 「気になる」保存 | ログイン済みユーザーが1タップでコンテンツをコレクションに追加 |
| カテゴリ分け | 「行きたい場所」「欲しいもの」「気になるスポット」等のカテゴリにコンテンツを分類 |
| コレクション一覧 | カテゴリ別サムネイル表示。タップで詳細画面へ |
| 「気になる」解除 | 保存済みコンテンツを削除 |

### 3. コンテンツ管理（管理者側）
| 機能 | 詳細 |
|------|------|
| コンテンツ投稿 | 管理者がタイトル・画像URL・説明文・外部リンク・ジャンルタグを登録 |
| コンテンツ編集・削除 | 登録済みコンテンツの更新・削除 |
| 人数集計の確認 | 各コンテンツの「気になる」保存数の確認 |

### 4. ユーザー管理・認証
| 機能 | 詳細 |
|------|------|
| 匿名閲覧 | アカウントなしでフィードの閲覧のみ可能。コレクションは使用不可 |
| アカウント登録 | メールアドレス＋パスワードで登録（Clerk使用） |
| ログイン・ログアウト | セッション管理（自動ログイン維持） |
| ジャンル設定 | 初回オンボーディングおよび設定画面でジャンルを選択・変更 |

## 非機能要件

### パフォーマンス
| 要件 | 目標値 | 根拠 |
|------|--------|------|
| ホームフィード初回表示 | 3秒以内 | 就寝前・通勤中の利用を想定。ストレスを感じさせない閾値 |
| 画像読み込み | 遅延読み込み（Lazy Load）対応 | 大量の画像を含むフィードでのスクロール性能確保 |
| API レスポンス | 500ms以内（P95） | 初期ユーザー数百人規模での目標値 |

### セキュリティ
- コレクションデータは本人のみ閲覧可（RLS: Row Level Security by Supabase）
- 「〇人が気になっています」は集計値のみ。個人を特定できる情報は非公開
- 認証はClerkに委譲（パスワードハッシュ・セッション管理）
- 管理者ページはIP制限または管理者ロール認証で保護

### スケーラビリティ
- MVPは数百人規模。Phase 2以降で数万人規模に対応
- Vercel（エッジネットワーク）+ Supabase（PostgreSQL）構成でスケールアウト可能
- 画像はCDN配信（Vercel Image Optimization or Cloudflare Images）

### アクセシビリティ・対応環境
- レスポンシブデザイン：スマートフォン（375px〜）・タブレット（768px〜）・PC（1280px〜）
- 主要ブラウザ対応：Chrome・Safari・Firefox（最新2バージョン）
- iOS Safari・Android Chrome対応

## 技術スタック

### 選定スタック

| レイヤー | 技術 | 選定理由 |
|----------|------|----------|
| フロントエンド | Next.js 14（App Router）+ TypeScript | SSR/SSGによる高速初期表示。画像最適化・レスポンシブ対応が標準。TypeScriptで型安全 |
| スタイリング | Tailwind CSS | 素早いUI実装。レスポンシブ対応が容易。デザイントークン管理しやすい |
| バックエンド | Next.js App Router（API Routes） | フロントと同一リポジトリで管理。サーバーレス関数で小規模スタートに最適 |
| データベース | Supabase（PostgreSQL） | RLSによる行レベルセキュリティ。リアルタイム購読（将来対応）。無料枠でMVPを低コストで開始可能 |
| 認証 | Clerk | 実装コストが低い。メール認証・ソーシャルログイン対応。UIコンポーネント標準提供 |
| 画像ストレージ | Supabase Storage or Cloudflare Images | コンテンツ画像の管理とCDN配信 |
| ホスティング | Vercel | Next.jsとの親和性最高。エッジネットワーク。プレビューデプロイで開発効率向上 |

### 将来的な追加技術（Phase 2以降）
- AIレコメンド：OpenAI API or ベクトルDB（pgvector）によるフィードパーソナライズ
- PWA対応：next-pwa によるオフラインキャッシュ・ホーム画面追加

## 開発ロードマップ

### Phase 1：MVP（〜3ヶ月）※ソロ開発想定
**目標：** コア機能の動作確認と初期ユーザー100人への提供

| 優先度 | タスク |
|--------|--------|
| 高 | プロジェクトセットアップ（Next.js + Supabase + Clerk + Vercel） |
| 高 | データベース設計・マイグレーション |
| 高 | 管理者用コンテンツ投稿機能 |
| 高 | ホームフィード表示（カード型・ジャンルフィルター） |
| 高 | カード詳細画面・外部リンク遷移 |
| 高 | 「気になる」保存・コレクション管理 |
| 高 | 認証（メール登録・ログイン）・匿名閲覧 |
| 高 | 「〇人が気になっています」表示 |
| 高 | レスポンシブデザイン対応 |
| 中 | 初回オンボーディング（ジャンル選択） |

**根拠：** ソロ開発のため、ビジネス仮説の検証に必要な最小限の機能に絞る。コンテンツは本人が手動キュレーション。Supabase + Clerk + Vercelの組み合わせでバックエンド実装コストを最小化し、フロントエンドの体験品質に集中する。

### Phase 2：継続率向上（3〜5ヶ月目）
**目標：** 継続率30%達成・フィードの自動パーソナライズ開始

- フィードのAIパーソナライズ（閲覧・保存の傾向学習）
- プッシュ通知・新着通知
- コレクションへのメモ追加機能
- ダークモード対応
- PWA対応（オフラインキャッシュ）

**根拠：** MVPで継続利用のフックを確認後、長期的な習慣化を促す機能を追加する。

### Phase 3：マネタイズ・拡張（6ヶ月以降）
**目標：** 収益モデルの確立・ジャンル拡張

- 提携スポット・商品への予約・購入導線（アフィリエイト）
- ジャンル追加（ゴルフ・音楽・映画等）
- コレクションのエクスポート機能
- 法人向け掲載プラン

## リスク分析

### 技術的リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| コンテンツ画像の読み込み遅延 | 高 | Vercel Image Optimization + Lazy Load。WebP変換 |
| Supabase無料枠の上限超過 | 中 | MVPで使用量をモニタリング。超過前にProプランへ移行 |
| 外部リンク先のコンテンツ消滅 | 中 | リンク切れ検知バッチを定期実行。スナップショットキャッシュの検討 |

### ビジネス的リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| 「見るだけ」で離脱が多い | 高 | コレクション機能を早期に訴求。「あなたのコレクション」として個人化を強調 |
| コンテンツ量不足によるフィードの空洞化 | 高 | リリース前に最低100件のコンテンツを手動で用意してから公開 |
| Pinterestとの差別化不足 | 中 | 「40代男性・日本語・ライフスタイル特化」のニッチさで差別化。Pinterestは英語コンテンツ比率が高い |

### 運用的リスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| コンテンツキュレーションの継続コスト | 中 | 将来的にユーザー投稿（厳選）または提携メディアのRSS自動取り込みで自動化 |
| 不適切コンテンツの混入 | 低 | 管理者のみ投稿可能（MVP時点）。将来は審査フロー導入 |

### 法的・コンプライアンスリスク
| リスク | 影響度 | 対策 |
|--------|--------|------|
| 外部コンテンツの著作権問題 | 中 | 画像をコピーせずリンクのみ保持。サムネイルはOGP画像を使用（フェアユース範囲） |
| 個人情報保護（プライバシーポリシー） | 中 | Clerkのデータ処理規約確認。プライバシーポリシーページを必ず実装 |

## UI/UXの方向性

- **デザインコンセプト：** 静かな上質感（Quiet Luxury）。余白を活かし、コンテンツ画像が映えるミニマルデザイン
- **トーン＆マナー：** 落ち着いたプロフェッショナル。騒がしさを排除。雑誌的な世界観
- **ターゲットデバイス：** モバイルファーストで設計。タブレット・PCでは2〜3カラムグリッドに拡張
- **カラーパレット：** オフホワイト背景・ディープグリーン or チャコールグレーをメイン。アクセントにアースブラウン
- **参考サイト/アプリ：**
  - [Pinterest](https://www.pinterest.jp/) ── ビジュアルフィードのレイアウト・コレクション体験
  - [Monocle](https://monocle.com/) ── 落ち着いた上質なビジュアルと世界観
  - [YAMAP](https://yamap.com/) ── 趣味特化SNSの日本語UX・コンテンツカード設計
