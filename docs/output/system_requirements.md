# システム要件定義書

## システム概要

**MITORI（読むだけSNS）** は、40代男性のSNS未使用者を主なターゲットとする、完全受信型のライフスタイルビジュアルフィードアプリである。「発信しなくていい」をコアコンセプトとし、投稿・フォロー・いいねといったSNS的な義務を一切排除する。ユーザーはキャンプ・温泉・サウナ・バイク・釣り・古着・ウイスキー・国内旅行などのジャンルを選択するだけで、自分のためのビジュアルフィードが生成され、気になったコンテンツをコレクションに貯めていける。管理者（開発者本人）がコンテンツをキュレーションして掲載し、ユーザーの反応データをもとに改善サイクルを高速で回すPOCとして開発する。

1. **システムの目的**
   - 解決する課題：40代男性SNS未使用者が発信プレッシャーなしに趣味の良質なビジュアル情報を受信できる場を提供
   - 目標とする成果：週1回以上の継続利用とコレクション機能による習慣化
   - 期待されるビジネスインパクト：趣味関連スポット・商品への送客による提携収益の創出

2. **主な機能**
   - コア機能：ビジュアルフィード表示、ジャンルフィルター、キーワード・場所検索、「気になる」保存、コレクション管理
   - 差別化機能：投稿機能の完全排除、「〇人が気になっています」の人数表示のみの社会性、外部コンテンツへのシームレスな導線
   - 管理機能：OGPスクレイピング＋手動入力によるコンテンツ登録、掲載可否の明示的な選択、ユーザー反応の分析

3. **提供する価値**
   - ユーザーへの価値：発信プレッシャーゼロで趣味の情報収集ができる。自分だけのコレクションが育つ充実感
   - ビジネスへの価値：趣味関連企業との送客提携、購買力のある40代男性層へのリーチ
   - 社会的価値：SNS疲れ・デジタルウェルビーイングへの貢献

4. **実現方法の概要**
   - 主要テクノロジー：Next.js (App Router) + TypeScript + Supabase + Clerk + Vercel
   - 重要なインテグレーション：OGPスクレイピングAPI、外部コンテンツリンク管理、将来的な地図API連携
   - スケーラビリティ：Vercel + Supabaseによるサーバーレス構成でスケールアウトに対応

**確定情報：**
- 開発体制：ソロ（1名）。設計・実装・コンテンツキュレーション・運用をすべて自身が担当
- 技術習熟度：Next.js・TypeScript・Supabase等は初心者。学習コストを考慮した設計が必要
- 予算：制約なし。ただしコスト最小化を優先（無料枠最大活用）
- 開発方針：POCとして高速にサイクルを回す。完成度より検証速度を重視
- サムネイル：OGPスクレイピング＋手動入力の両方に対応
- 掲載管理：管理者が明示的に公開可否を選択できる仕組みが必須

**置いた仮定：**
- [仮] MVPのコンテンツは手動キュレーション（開発者本人が投稿）で運用。AIレコメンドはPhase 2以降
- [仮] 初期ユーザー規模はMVP時に数百人、1年後に数万人を想定
- [仮] 収益化はPhase 2〜3。MVPではマネタイズより継続率の検証を優先
- [仮] 地図連携（Google Maps等）はPhase 2以降。MVPでは場所名・都道府県のテキスト検索に留める

## 機能要件

### 1. コンテンツフィード（ユーザー向け）

| 機能 | 詳細 |
|------|------|
| ホームフィード表示 | カード型グリッドレイアウト。画像・タイトル・ジャンルタグを表示。無限スクロール対応 |
| ジャンルフィルター | キャンプ・温泉・サウナ・バイク・釣り・古着・ウイスキー・国内旅行の8ジャンル以上。ホームフィード上部のタブで切り替え |
| キーワード・場所検索 | 場所名・都道府県・キーワードでフィード内コンテンツを検索 |
| カード詳細画面 | 大画像・説明文・出典元・場所情報・「〇人が気になっています」・「気になる」ボタン・外部リンクボタンを表示 |
| 外部リンク遷移 | X・note・YouTube等へのリンクを表示。新規タブで外部サイトへ遷移 |
| 「〇人が気になっています」 | コンテンツごとの保存人数の集計値を表示。個人特定不可 |

### 2. コレクション管理（ユーザー向け）

| 機能 | 詳細 |
|------|------|
| 「気になる」保存 | ログイン済みユーザーが1タップでコンテンツをコレクションに追加 |
| カテゴリ分け | 「行きたい場所」「欲しいもの」「気になるスポット」等のカテゴリにコンテンツを分類 |
| コレクション一覧 | カテゴリ別サムネイル表示。タップで詳細画面へ |
| 「気になる」解除 | 保存済みコンテンツを削除 |

### 3. コンテンツ管理（管理者向け）

| 機能 | 詳細 |
|------|------|
| OGPスクレイピング | URLを入力するとOGP画像・タイトル・説明文を自動取得してフォームに反映 |
| サムネイル手動入力 | 画像ファイルのアップロードまたは画像URLの直接入力（スクレイピング失敗時の代替） |
| コンテンツ登録・編集・削除 | タイトル・画像・説明文・外部リンク・ジャンルタグ・場所名・都道府県を登録・編集 |
| 掲載ステータス管理 | 下書き／公開／非公開の切り替え。公開するコンテンツを管理者が明示的に選択 |
| 一覧・フィルター | 登録済みコンテンツをステータス・ジャンルで絞り込んで確認 |

### 4. ユーザー反応の分析（管理者向け）

| 機能 | 詳細 |
|------|------|
| コンテンツ別集計 | 各コンテンツの「気になる」保存数・閲覧数を一覧表示 |
| ジャンル別傾向 | どのジャンルのコンテンツが保存されやすいかの集計 |
| 人気コンテンツランキング | 保存数・閲覧数の多い順にソートして表示 |

### 5. ユーザー管理・認証

| 機能 | 詳細 |
|------|------|
| 匿名閲覧 | アカウントなしでフィードの閲覧・検索のみ可能。コレクションは使用不可 |
| アカウント登録 | メールアドレス＋パスワードで登録（Clerk使用） |
| ログイン・ログアウト | セッション管理（自動ログイン維持） |
| ジャンル設定 | 初回オンボーディングおよび設定画面でジャンルを選択・変更 |

## 非機能要件

### パフォーマンス

| 要件 | 目標値 | 根拠 |
|------|--------|------|
| ホームフィード初回表示 | 3秒以内 | 就寝前・通勤中の利用を想定。ストレスを感じさせない閾値 |
| 画像読み込み | 遅延読み込み（Lazy Load）対応 | 大量の画像を含むフィードでのスクロール性能確保 |
| OGPスクレイピング | 5秒以内にレスポンス | 管理者の入力体験を損なわない範囲 |
| APIレスポンス | 500ms以内（P95） | 初期ユーザー数百人規模での目標値 |

### セキュリティ
- コレクションデータは本人のみ閲覧可（RLS: Row Level Security by Supabase）
- 「〇人が気になっています」は集計値のみ表示。個人を特定できる情報は非公開
- 認証はClerkに委譲（パスワードハッシュ・セッション管理）
- 管理者ページは管理者ロール認証で保護（Clerkのロール機能を利用）

### コスト方針（無料枠優先）

| サービス | 無料枠 | 超過時の対応 |
|----------|--------|-------------|
| Vercel | Hobbyプラン（無料）| POC段階では無料枠内に収める |
| Supabase | 無料プラン（500MB DB・1GB Storage）| ユーザーが増えたらProプラン（$25/月）へ移行 |
| Clerk | 無料プラン（月10,000 MAU）| POC段階では無料枠内に収める |
| OGPスクレイピング | Next.js API Routeで自前実装（無料） | 外部APIは使わず自前で実装 |

### アクセシビリティ・対応環境
- レスポンシブデザイン：スマートフォン（375px〜）・タブレット（768px〜）・PC（1280px〜）
- 主要ブラウザ対応：Chrome・Safari・Firefox（最新2バージョン）
- iOS Safari・Android Chrome対応

## 技術スタック

### 選定スタック（初心者・コスト最小・POC重視で選定）

| レイヤー | 技術 | 選定理由 |
|----------|------|----------|
| フロントエンド | Next.js 14（App Router）+ TypeScript | SSR/SSGによる高速初期表示。Vercelとの親和性最高。学習リソースが豊富 |
| スタイリング | Tailwind CSS | クラス名だけでUIが作れる。初心者でも素早くレスポンシブ対応が可能 |
| UIコンポーネント | shadcn/ui | Tailwind CSS + Radix UIベースの高品質コンポーネント集。コピペで使える |
| バックエンド | Next.js App Router（Server Actions + API Routes） | フロントと同一リポジトリ。サーバーレスで運用コストゼロ |
| データベース | Supabase（PostgreSQL） | RLSで行レベルセキュリティ。ダッシュボードでDB管理が容易。無料枠あり |
| 認証 | Clerk | 実装コストが低い。UIコンポーネント標準提供。Next.jsとの統合が簡単 |
| OGPスクレイピング | cheerio + node-fetch（自前実装） | 外部API不要。Next.js API Routeで実装。無料 |
| 画像ストレージ | Supabase Storage | DBと同じサービスで管理が一元化。無料枠内で開始可能 |
| ホスティング | Vercel | Next.jsとの親和性最高。Hobbyプランは無料。プレビューデプロイ対応 |

### 将来的な追加技術（Phase 2以降）
- AIレコメンド：pgvector（Supabase）によるベクトル検索でフィードパーソナライズ
- 地図連携：Google Maps API or Mapbox
- PWA対応：next-pwa によるオフラインキャッシュ・ホーム画面追加

## 開発ロードマップ

### Phase 1：POC・MVP（〜3ヶ月）※ソロ・初心者想定

**目標：** 動くものを素早く作り、身近な40代男性10〜20人に使ってもらいフィードバックを得る

| 週 | タスク |
|----|--------|
| 1〜2週目 | 環境構築（Next.js + Supabase + Clerk + Vercel）、DB設計・マイグレーション |
| 3〜4週目 | 管理者側：OGPスクレイピング＋コンテンツ登録・掲載管理 |
| 5〜6週目 | ユーザー側：ホームフィード・ジャンルフィルター・カード詳細 |
| 7〜8週目 | 「気になる」保存・コレクション管理・認証（Clerk） |
| 9〜10週目 | キーワード・場所検索・「〇人が気になっています」表示 |
| 11〜12週目 | レスポンシブデザイン調整・コンテンツ100件登録・初回リリース |

**根拠：** 初心者のソロ開発のため、1週間2〜3タスクのペースで進める。shadcn/ui + TailwindでUI実装コストを下げ、Supabase + Clerkでバックエンド実装を最小化する。

### Phase 2：継続率向上（4〜6ヶ月目）
- 分析ダッシュボード強化（どの写真が好まれるかの可視化）
- 地図連携（「行きたい場所」をマップ表示）
- フィードのパーソナライズ（保存傾向から学習）
- プッシュ通知・ダークモード・PWA対応

### Phase 3：マネタイズ・拡張（7ヶ月以降）
- 提携スポット・商品への予約・購入導線
- ジャンル追加（ゴルフ・音楽・映画等）
- 法人向け掲載プラン

## リスク分析

### 技術的リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| Next.js学習コストによる開発遅延 | 高 | shadcn/ui + Vercelのテンプレートから始める。公式ドキュメント・YouTubeを活用 |
| OGPスクレイピングが取得できないサイトがある | 中 | 手動入力で補完できる設計にする（必須ではなく補助機能として位置づけ） |
| Supabase無料枠の上限超過 | 中 | MVPで使用量をモニタリング。超過前にProプランへ移行（$25/月） |
| 外部リンク先のコンテンツ消滅 | 低 | リンク切れを定期チェックするバッチを将来実装 |

### ビジネス的リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| 「見るだけ」で離脱が多い | 高 | コレクション機能を早期に訴求。「あなたのコレクション」として個人化を強調 |
| コンテンツ量不足によるフィードの空洞化 | 高 | リリース前に最低100件のコンテンツを手動で用意してから公開 |
| Pinterestとの差別化不足 | 中 | 「40代男性・日本語・ライフスタイル特化・場所検索」のニッチさで差別化 |

### 運用的リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| ソロ開発によるコンテンツ更新の停滞 | 中 | OGPスクレイピングで登録工数を下げる。週10件程度の更新を習慣化 |
| 不適切コンテンツの混入 | 低 | 管理者のみ投稿可能（MVP時点）。将来は審査フロー導入 |

### 法的・コンプライアンスリスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| OGP画像の著作権問題 | 中 | 画像をコピーせずリンクのみ保持。サムネイルはOGP画像のURLを参照（直リンク）。利用規約に記載 |
| 個人情報保護（プライバシーポリシー） | 中 | Clerkのデータ処理規約確認。プライバシーポリシーページを必ず実装 |

## UI/UXの方向性

- **デザインコンセプト：** 静かな上質感（Quiet Luxury）。余白を活かし、コンテンツ画像が映えるミニマルデザイン
- **トーン＆マナー：** 落ち着いたプロフェッショナル。騒がしさを排除。雑誌的な世界観
- **ターゲットデバイス：** モバイルファーストで設計。タブレット・PCでは2〜3カラムグリッドに拡張
- **カラーパレット：** オフホワイト背景・ディープグリーン or チャコールグレーをメイン。アクセントにアースブラウン
- **参考サイト/アプリ：**
  - [Pinterest](https://www.pinterest.jp/) ── ビジュアルフィードのレイアウト・コレクション体験
  - [Monocle](https://monocle.com/) ── 落ち着いた上質なビジュアルと世界観
  - [YAMAP](https://yamap.com/) ── 趣味特化SNSの日本語UX・コンテンツカード設計
