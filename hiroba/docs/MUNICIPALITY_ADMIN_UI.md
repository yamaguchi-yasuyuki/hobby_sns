# 自治体（施設登録担当者）向け管理画面 — 画面設計

対象: Clerk `hirobaAccountKind: "municipality_staff"` で入域する **`/admin/municipality`** 配下。  
一般ユーザー向けフィード（`/`）とは別レイアウト（ボトムナビ非表示・管理用サブナビ）。

## 1. ゴールと権限

- **一覧**: 自組織が登録した施設カードの一覧表示（公開・下書きの状態が分かる）。
- **新規登録 / 編集**: 同一フォームで、下記フィールドを入力・更新。
- **削除**: 確認ダイアログのうえ論理削除または物理削除（実装は DB 方針に合わせる）。
- **権限**: 当面は「職員＝1 自治体」の想定。将来は `organization_id` で RLS し、他自治体データは見えないようにする。

## 2. 情報設計（登録項目 ↔ 既存型）

| 画面セクション | フィールド例 | 型（`Facility` 想定） | 一般利用者向けの意味 |
|----------------|--------------|----------------------|----------------------|
| イメージ | メイン写真 URL（将来: アップロード） | `image_url`, `images[]` | カード・詳細のビジュアル |
| テキスト | タイトル、短い説明、詳細本文 | `title`, `description`, `body?` | 施設の紹介文 |
| リンク | 公式・地図・予約等（複数） | `links[]`（`label`, `url`, `type`） | 外部サイトへの導線 |
| 利用情報 | 料金、営業時間、所在地、都道府県 | `admission_fee?`, `opening_hours?`, `location_name`, `prefecture` | 訪問判断に必要な情報 |
| 分類・運営 | カテゴリ、運営主体名、主体種別 | `category`, `operator`, `operator_type` | フィルタ・バッジ表示 |
| 公式 URL | 自治体・施設の公式ページ | `official_url` | 一覧・詳細の「公式」リンク元 |
| 公開制御 | 下書き / 公開 / 非公開 | `status` | 一般フィードへの掲載可否 |

※ 画像は **Phase 1** では URL 入力、**Phase 2** で Supabase Storage + 署名付きアップロードを想定。

## 4. フォーム構成（ワイヤーフレーム）

1. **ヘッダー**: 「施設を登録」「施設を編集」＋ ステータス（下書き保存 / 公開予定）。
2. **画像**: メイン画像 URL、追加画像（複数行テキスト or リスト UI）。将来「ファイルを選択」ボタンを同一ブロックに追加。
3. **テキスト**: タイトル（必須）、一覧用説明、詳細本文（リッチテキストは後続）。
4. **リンク**: 行追加で「表示ラベル / URL / 種別」。公式・地図・予約等。
5. **利用者向け情報**: 料金、営業時間、施設名（所在地ラベル）、都道府県。
6. **分類・運営**: カテゴリ（プルダウン）、運営主体名、主体種別（市営など）。
7. **フッターアクション**: 「下書き保存」「公開する」「キャンセル（一覧へ）」。

## 5. 削除フロー

- 一覧の「削除」→ モーダル: 施設名の再表示 + 「削除すると一般ユーザーから見えなくなります」。
- 推奨: 最初は **論理削除**（`status = unpublished` または `deleted_at`）で監査に備える。

## 6. 技術メモ（実装フェーズ）

- **現状**: UI スケルトンは `mockFacilities` を表示。保存・削除 API は未接続（ボタンで案内のみ可）。
- **次**: `facilities` テーブル（`organization_id`, `created_by`, `status`, 上記カラム）と Storage バケット、RLS（職員は自組織のみ CRUD）。
- **認証**: 既存どおり Clerk + `municipality_staff`。組織紐付けは Clerk Organization または `users.municipality_id` 等で拡張。
